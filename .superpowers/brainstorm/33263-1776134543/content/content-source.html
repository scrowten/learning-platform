Where does content live?

API-first means something has to be the source of truth. Three architectures — pick the one that fits how you'll author content.

A

Git → Parsed → API (JAMstack)

Markdown + notebooks stay in this repo. A build/sync job parses frontmatter, extracts metadata, and loads it into a database. API serves from DB. Authoring = git commit.

markdown files → parser/indexer → postgres/sqlite → FastAPI → React/Flask webapp

Pros

  • Git is source of truth (version control, PRs)
  • Human-readable content
  • Works offline, no external dependencies

Cons

  • Need sync/build pipeline
  • Schema lives in frontmatter — discipline required
B

Database-First

Content authored directly into a structured DB (via admin UI or scripts). API serves live from DB. Markdown is optional export format. Tightly integrated with agentic-ai's existing PostgreSQL schema.

admin UI / scripts → postgres → FastAPI → React/Flask webapp

Pros

  • Cleanest API layer
  • Rich relational queries (prerequisites, cross-domain links)
  • AI features plug in naturally

Cons

  • Harder to author/edit (no plain text files)
  • Lose git history for content
C

Git as Source of Truth + DB as Index

Markdown/notebooks in git = write path. DB = read/query path. Sync runs on push. Best of both: human-friendly authoring, machine-friendly querying. This is how Docusaurus + Algolia works, applied to an API.

git push → CI sync job → postgres (index) → FastAPI → webapp / agentic-ai
↑ markdown files stay readable, DB stays queryable

Pros

  • Git history for content
  • Rich API querying
  • Aligns with agentic-ai pipeline
  • Most extensible

Cons

  • Sync pipeline to maintain
  • Most moving parts upfront

My recommendation: C — git stays human-friendly, DB enables the AI features.
